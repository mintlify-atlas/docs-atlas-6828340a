## Get started in four steps

This guide will take you from zero to running your first AI coding task in an isolated sandbox.

<Note>
  **Prerequisites:** Python 3.11+, Docker Desktop (or Docker Engine on Linux), and an Anthropic API key from [console.anthropic.com](https://console.anthropic.com)
  
  On Linux, also install rsync: `apt install rsync`
</Note>

### Step 1: Install FTL

<CodeGroup>
```bash Install from GitHub
git clone https://github.com/vvennela/ftl
cd ftl
pip install -e .
```

```bash Verify installation
ftl --version
```
</CodeGroup>

<Tip>
Use a virtual environment to avoid conflicts:

```bash
python -m venv venv
source venv/bin/activate  # On Windows: venv\Scripts\activate
pip install -e .
```
</Tip>

### Step 2: Run setup wizard

The one-time setup wizard configures your agent, tester model, and API credentials:

```bash
ftl setup
```

You'll be prompted to:

<Steps>
  <Step title="Choose your agent">
    Select which AI coding agent to use:
    
    ```
    Which agent do you want to use?
      1. Claude Code  (Anthropic, recommended)
      2. Codex        (OpenAI)
      3. Aider        (open-source)
      4. Kiro         (AWS)
      Choice [1]:
    ```
    
    The wizard pulls the appropriate Docker image from `vvenne/ftl:<tag>`
  </Step>
  
  <Step title="Choose test generation model">
    Select the LLM model for adversarial test generation:
    
    ```
    Which model for test generation?
      1. Anthropic API — claude-haiku  (uses ANTHROPIC_API_KEY)
      2. AWS Bedrock   — claude-sonnet (uses AWS credentials)
      3. Skip test generation
      Choice [1]:
    ```
    
    Tests generate in parallel while the agent codes, so there's no added latency
  </Step>
  
  <Step title="Save your API key">
    If you haven't set `ANTHROPIC_API_KEY` yet:
    
    ```
    Anthropic API key
      Get one at https://console.anthropic.com
      ANTHROPIC_API_KEY: sk-ant-...
    ```
    
    Your key is saved to `~/.ftl/credentials` (mode 600) and loaded automatically on every invocation
  </Step>
</Steps>

Your choices are saved globally to `~/.ftl/config.json` and used as defaults for every new project.

<Warning>
  **Docker Hub images** are pulled automatically:
  
  - `vvenne/ftl:latest` — Claude Code
  - `vvenne/ftl:codex` — Codex
  - `vvenne/ftl:aider` — Aider
  - `vvenne/ftl:kiro` — Kiro
  - `vvenne/ftl:full` — All agents
  
  If the pull fails, FTL builds the image locally from your cloned Dockerfile.
</Warning>

### Step 3: Initialize your project

Navigate to your project directory and create a `.ftlconfig`:

```bash
cd your-project
ftl init
```

This creates `.ftlconfig` with values from your global setup:

```json .ftlconfig
{
  "agent": "claude-code",
  "tester": "claude-haiku-4-5-20251001"
}
```

<Tip>
  Edit `.ftlconfig` to override settings per-project. See [Configuration reference](/configuration/ftlconfig) for all available fields.
</Tip>

### Step 4: Run your first task

```bash
ftl code 'create a Stripe payment module'
```

<Steps>
  <Step title="Snapshot">
    FTL snapshots your project to `~/.ftl/snapshots/<id>` using rsync
  </Step>
  
  <Step title="Boot sandbox">
    Docker container starts (or reuses existing) with shadow credentials injected
  </Step>
  
  <Step title="Agent + tests run in parallel">
    Claude Code writes code while adversarial tests generate simultaneously
    
    ```
    Running agent  (generating tests in parallel via claude-haiku-4-5-20251001)
    ```
  </Step>
  
  <Step title="Review and approve">
    FTL shows you a diff of all changes. You can:
    
    - Type `a` — approve and merge changes to your project
    - Type `r` — reject and discard all changes
    - Type any question — ask the model about the diff (e.g. "does this handle null inputs?")
    
    ```
    Review diff and approve or reject changes:
      a — approve and merge
      r — reject and discard
      or ask a question about the changes
    
    Your choice >
    ```
  </Step>
</Steps>

<Note>
  Changes only touch your project after you type `a` and confirm. The agent never has direct write access to your filesystem.
</Note>

## What just happened?

Let's break down the FTL workflow:

<AccordionGroup>
  <Accordion icon="camera" title="Snapshot creation">
    FTL uses rsync to copy your project state to `~/.ftl/snapshots/<id>`. Files matching `.ftlignore` patterns are excluded (similar to `.gitignore`).
    
    The snapshot is mounted read-only into the Docker container at `/workspace`.
  </Accordion>
  
  <Accordion icon="key" title="Shadow credentials">
    If you have a `.env` file in your project root:
    
    ```bash .env
    STRIPE_SECRET_KEY=sk_live_abc123
    OPENAI_API_KEY=sk-proj-xyz789
    ```
    
    FTL generates shadow values and injects them into the sandbox:
    
    ```bash Sandbox environment
    STRIPE_SECRET_KEY=ftl_shadow_stripe_secret_key_7f8a2b3c
    OPENAI_API_KEY=ftl_shadow_openai_api_key_4d9e2a1f
    ```
    
    Your real keys **never enter the container**. The agent writes code using shadow values.
  </Accordion>
  
  <Accordion icon="vial" title="Parallel test generation">
    While the agent codes, FTL sends your task description to the tester model with a prompt like:
    
    ```
    Generate adversarial tests for this task:
    "create a Stripe payment module"
    
    Write pytest test cases that verify:
    - Correct implementation
    - Error handling
    - Edge cases
    ```
    
    Tests run immediately when the agent finishes, catching issues before you review.
  </Accordion>
  
  <Accordion icon="magnifying-glass" title="Credential linting">
    Before showing the diff, FTL scans for:
    
    - Hardcoded shadow values (`ftl_shadow_*` pattern)
    - Known credential patterns (Stripe keys, AWS keys, GitHub tokens, etc.)
    - Dangerous SQL operations (`DROP TABLE`, `TRUNCATE` without `WHERE`)
    - Dangerous shell commands (`rm -rf`, `chmod -R 777`)
    
    Findings are displayed as warnings but never block review (unless you configure Bedrock Guardrails).
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion icon="triangle-exclamation" title="Docker not found or not running">
    ```bash
    [red]Docker not found. Install Docker Desktop and try again.[/red]
    ```
    
    **Solution:** Install Docker Desktop from [docs.docker.com/get-docker](https://docs.docker.com/get-docker/) and ensure it's running.
  </Accordion>
  
  <Accordion icon="triangle-exclamation" title="No .ftlconfig found">
    ```bash
    [red]No .ftlconfig found. Run 'ftl init' first.[/red]
    ```
    
    **Solution:** Navigate to your project directory and run `ftl init`.
  </Accordion>
  
  <Accordion icon="triangle-exclamation" title="ANTHROPIC_API_KEY is not set">
    ```bash
    [bold red]ANTHROPIC_API_KEY is not set.[/bold red]
      Run: ftl auth ANTHROPIC_API_KEY <your-key>
    ```
    
    **Solution:** Save your API key:
    
    ```bash
    ftl auth ANTHROPIC_API_KEY sk-ant-...
    ```
  </Accordion>
  
  <Accordion icon="triangle-exclamation" title="Image pull fails">
    If `ftl setup` can't pull from Docker Hub, it falls back to building locally. This requires the Dockerfile in your cloned `ftl` repository.
    
    If the build fails, check that you cloned the full repository and didn't just install via pip.
  </Accordion>
</AccordionGroup>
